# 4.4 步骤：Mac Mini TTS 和音频回传验证

## 目标

记录 Mac Mini 把 Hermes 的文本回答转换成音频，并通过 TTS URL 让小爱音箱 Pro LX06 播放的已验证结果。

目标链路：

```text
Hermes answer text
  -> Mac Mini TTS
  -> playable audio URL
  -> open-xiaoai server/client
  -> 音箱扬声器播放
```

验证时可以先用固定文本，不依赖 STT 和 Hermes 实时调用；当前端到端链路中已经使用 TTS URL 播放。

## 为什么单独验证 TTS

4-* 最终希望 Hermes 完全接管音箱。接管不只是“能听懂”，还必须“能说出来”。

3-* 路线目前使用：

```text
speaker.play(text=answer)
```

这通常仍然复用音箱/小米侧文字播报能力。

4-* 已验证的是：

```text
Mac Mini 生成音频
  -> TTS URL
  -> 音箱播放这段音频
```

这样可以：

- 降低对小米 TTS 的依赖。
- 控制音色、语速和稳定性。
- 支持未来流式 TTS。
- 避免小米云回答和 Hermes 回答抢播。

## 已采用的推进顺序

1. Mac Mini 生成本地 wav/mp3 文件。
2. 在 Mac Mini 本机播放确认音频正常。
3. 通过 open-xiaoai 让音箱播放本地/远端音频。
4. 验证音箱播放期间的打断和状态恢复。
5. 再接 Hermes 动态文本。

## TTS 引擎选择

第一阶段可以任选一个稳定方案：

```text
edge-tts
macOS say
OpenAI-compatible TTS
本地 TTS 服务
```

选择标准：

- 能快速生成中文语音。
- 输出 wav/mp3 可控。
- 延迟可测。
- 不阻塞 server 主循环。

第一阶段不追求最好音色，先追求可控播放。

## 固定测试文本

先用固定文本：

```text
你好，我是 Hermes，现在正在通过本地语音链路接管这个音箱。
```

再用较长文本：

```text
今天适合去附近的公园走一走。如果天气不错，可以带孩子晒晒太阳，顺便准备一点水和小零食。
```

## 输出音频格式

建议第一阶段统一：

```text
wav
24000 Hz or 16000 Hz
mono
16-bit PCM
```

如果音箱播放接口更适合 mp3，则记录转换链路：

```text
text -> mp3 -> 音箱播放
```

输出目录：

```text
audio-samples/tts/
```

命名：

```text
YYYYMMDD-HHMMSS-tts.wav
```

## 已确认的音箱播放方式

验证过程中检查过：

```text
open-xiaoai/examples/xiaozhi/xiaozhi/services/speaker.py
open-xiaoai/packages/client-rust/src/services/audio/play.rs
open-xiaoai/packages/client-rust/src/services/connect/rpc.rs
open-xiaoai/examples/xiaozhi/src/server.rs
```

结论：

1. `speaker.play(text=...)` 仍作为 3-* legacy route 的文字播报能力保留。
2. 4-* 当前已验证路径是 Mac Mini 生成 TTS 音频并通过 URL 让音箱播放。
3. `Stream { tag: "play" }` 仍是后续低延迟流式播放方向，不是当前已验证主路径。
4. 音频播放期间的自然打断和状态恢复放到 `3-design` 中工程化。

## 推荐验证路径

### 路径 A：播放 URL

当前已验证路径：

```text
Mac Mini 生成 tts.wav
  -> Mac Mini 起本地 HTTP 静态服务
  -> 音箱播放 http://192.168.1.9:<port>/tts.wav
```

优点：简单，适合验证，当前已经可用。

缺点：首播延迟可能较高，不适合最终流式。

### 路径 B：WebSocket 音频流

后续如果 open-xiaoai 支持直接下发播放流：

```text
Mac Mini 生成 PCM/opus frames
  -> WebSocket stream tag=play
  -> 音箱播放
```

优点：适合低延迟和流式 TTS。

缺点：需要确认编码、分帧和播放缓冲。

### 路径 C：临时复用文本播报

3-* legacy route 中保留：

```text
speaker.play(text=answer)
```

这属于旧路线的文字播报能力，不是 4-* 的 TTS URL 主路径。

## 验收标准

4.4 完成标准：

- [x] Mac Mini 能把固定中文文本生成可播放音频。
- [x] 生成的音频在 Mac Mini 本机播放正常。
- [x] 音箱能通过 TTS URL 播放 Mac Mini 生成的音频。
- [x] 播放方式、音频格式、采样率已记录。
- [x] 播放失败时有明确错误日志。
- [x] 3-* 的 `speaker.play(text=...)` legacy route 不受影响。
