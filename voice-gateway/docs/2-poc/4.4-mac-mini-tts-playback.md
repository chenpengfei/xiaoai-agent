# 4.4 步骤：Mac Mini TTS 和音频回传验证

## 目标

记录 Mac Mini 把 Hermes 的文本回答转换成音频，并通过 TTS URL 让小爱音箱 Pro LX06 播放的已验证结果。

目标链路：

```text
Hermes answer text
  -> Mac Mini TTS
  -> playable audio URL
  -> open-xiaoai server/client
  -> 音箱扬声器播放
```

验证时可以先用固定文本，不依赖 STT 和 Hermes 实时调用；当前端到端链路中已经使用 TTS URL 播放。

## 为什么单独验证 TTS

4-* 最终希望 Hermes 完全接管音箱。接管不只是“能听懂”，还必须“能说出来”。

3-* 历史路线曾使用：

```text
speaker.play(text=answer)
```

这通常仍然复用音箱/小米侧文字播报能力；当前 `voice-gateway` 运行时不再使用这条 TTS 路径。

4-* 已验证的是：

```text
Mac Mini 生成音频
  -> TTS URL
  -> 音箱播放这段音频
```

这样可以：

- 降低对小米 TTS 的依赖。
- 控制音色、语速和稳定性。
- 支持未来流式 TTS。
- 避免小米云回答和 Hermes 回答抢播。

## 已采用的推进顺序

1. Mac Mini 生成本地 wav/mp3 文件。
2. 在 Mac Mini 本机播放确认音频正常。
3. 通过 open-xiaoai 让音箱播放本地/远端音频。
4. 验证音箱播放期间的打断和状态恢复。
5. 再接 Hermes 动态文本。

## TTS 引擎选择

当前运行时已固定为 Edge TTS：

```text
Edge TTS
  -> mp3
  -> Mac Mini HTTP URL
  -> 音箱 miplayer 播放
```

选择依据：

- `5-tts-quality-comparison.md` 验证 Edge TTS 听感明显好于小米原生 TTS。
- 本地模型实验的音色或速度未超过 Edge。
- Edge TTS 延迟可接受，运行时不再保留其他 TTS 引擎和 fallback；仅固定短提示进入文件缓存。

小米原生 `tts_play.sh` 只保留在历史验证记录中，不进入当前 `voice-gateway` 运行时。

## 固定测试文本

先用固定文本：

```text
你好，我是 Hermes，现在正在通过本地语音链路接管这个音箱。
```

再用较长文本：

```text
今天适合去附近的公园走一走。如果天气不错，可以带孩子晒晒太阳，顺便准备一点水和小零食。
```

## 输出音频格式

当前统一输出：

```text
mp3
24000 Hz
mono
```

生成和播放链路：

```text
text -> Edge TTS mp3 -> 音箱 miplayer 播放
```

输出目录：

```text
audio-samples/tts/
```

命名：

```text
YYYYMMDD-HHMMSS-<playback_id>.mp3
```

## 已确认的音箱播放方式

验证过程中检查过：

```text
open-xiaoai/examples/xiaozhi/xiaozhi/services/speaker.py
open-xiaoai/packages/client-rust/src/services/audio/play.rs
open-xiaoai/packages/client-rust/src/services/connect/rpc.rs
open-xiaoai/examples/xiaozhi/src/server.rs
```

结论：

1. 当前运行时只走 Mac Mini Edge TTS 生成 MP3，再通过 URL 让音箱播放；“已连接”等固定短提示可复用缓存 mp3。
2. `speaker.play(text=...)` / `tts_play.sh` 不再作为 `voice-gateway` 的 TTS 路径。
3. `Stream { tag: "play" }` 属于后续播放链路优化方向，不是当前 TTS 引擎替代方案。
4. 音频播放期间的自然打断和状态恢复放到 `3-design` 中工程化。

## 推荐验证路径

当前已验证路径：

```text
Mac Mini 生成 Edge TTS mp3
  -> Mac Mini 起本地 HTTP 静态服务
  -> 音箱播放 http://192.168.1.9:<port>/<file>.mp3
```

优点：简单，适合验证，当前已经可用。

缺点：首播延迟可能较高，不适合最终流式。


## 验收标准

4.4 完成标准：

- [x] Mac Mini 能把固定中文文本生成可播放音频。
- [x] 生成的音频在 Mac Mini 本机播放正常。
- [x] 音箱能通过 TTS URL 播放 Mac Mini 生成的音频。
- [x] 播放方式、音频格式、采样率已记录。
- [x] 播放失败时有明确错误日志。
- [x] 运行时不再依赖 `speaker.play(text=...)` / `tts_play.sh`。
