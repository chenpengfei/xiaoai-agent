# 4.3 步骤：Mac Mini STT 验证

## 目标

把 `4.2` 保存下来的音箱语音样本转换成文字，并记录当前已验证的实时 STT 链路。

目标链路：

```text
音箱 utterance.wav
  -> Mac Mini STT
  -> text
  -> normalized user question
```

本步骤的核心是验证 STT 的准确率、速度和错误处理。端到端 TTS 播放结果见 [4.4 Mac Mini TTS 和音频回传验证](./4.4-mac-mini-tts-playback.md)。

## 推荐推进顺序

推进顺序是先离线，再实时：

1. 离线样本 STT。
2. 批量样本准确率对比。
3. 封装 STT Python 函数。
4. 接入实时 utterance session。
5. 输出统一 STT result 结构。

PoC 已按这个顺序完成。后续工程化时仍应保留这个节奏，避免实时链路问题和模型问题混在一起排查。

## STT 引擎选择

当前已确定使用：

```text
sherpa-onnx
```

原因：

- Mac Mini 本地运行，不依赖云端 STT。
- 中文短句识别稳定，适合“小爱同学 / 你好 <问题>”这类指令。
- Python API 可直接嵌入 xiaozhi server。
- 已验证在 Mac Mini 上识别速度很快，约 44-52 ms 处理一段约 5.94 秒样本，RTF 约 0.007-0.009。
- 同一个库可同时使用 Paraformer STT 和 Silero VAD，方便后续实时 utterance 切分。

当前验证过的中文离线模型：

```text
csukuangfj/sherpa-onnx-paraformer-zh-2024-03-09
model.int8.onnx + tokens.txt
```

模型本地目录：

```text
/Users/chenpengfei/projects/vibe-coding/xiaoai-agent/models/sherpa-onnx-paraformer-zh-2024-03-09
```

xiaozhi uv 环境中已包含：

```text
sherpa-onnx==1.12.32
```

后续不要再把 faster-whisper / whisper.cpp / mlx-whisper 作为 4.3 主线；它们只作为备用方案记录。

## sherpa-onnx 调用方式

旧 xiaozhi 示例位于上游归档仓库：
<https://github.com/idootop/open-xiaoai/tree/main/examples/xiaozhi>

从该示例目录运行，确保使用项目 uv 环境：

```bash
PATH="/Users/chenpengfei/.local/bin:/opt/homebrew/bin:$PATH" uv run python ...
```

Python 构造方式：

```python
recognizer = sherpa_onnx.OfflineRecognizer.from_paraformer(
    paraformer=str(model),
    tokens=str(tokens),
    num_threads=4,
    sample_rate=16000,
    feature_dim=80,
    decoding_method="greedy_search",
    provider="cpu",
)
stream = recognizer.create_stream()
stream.accept_waveform(16000, samples_float32)
recognizer.decode_stream(stream)
text = stream.result.text.strip()
```

对当前音箱录音样本，STT 前加增益效果更好：

```bash
ffmpeg -i input.wav -af "volume=12dB" output.wav
```

旧 xiaozhi PoC 曾通过 `XIAOAI_LOCAL_STT_*` 环境变量调节实时 STT。当前 `voice-gateway`
主链路改用 `VOICE_GATEWAY_*` 配置，详见 `scripts/run-voice-gateway-minimal.sh`。

## 文本归一化

STT 输出后要做轻量归一化：

```text
去掉首尾空白
去掉明显标点噪声
中文全角/半角标点统一
保留原始 text 和 normalized_text
```

不要过度改写用户问题。

示例：

```text
raw:        你好，今天适合去哪玩？
normalized: 你好今天适合去哪玩
question:   今天适合去哪玩
```

如果 4-* 使用“你好”作为本地唤醒词，进入 Hermes 的问题可以去掉前缀：

```python
if normalized.startswith("你好"):
    question = normalized.removeprefix("你好").strip(" ，,。？?：:")
```

实际录音里可能包含小爱唤醒词、应答音或历史播放尾音，例如：

```text
小爱同学哎你好一加一等于几
```

因此本项目当前实现按最后一次 `你好` 截取问题：

```python
if "你好" in normalized_text:
    question = normalized_text.rsplit("你好", 1)[1].strip(" ，,。？?：:")
```

本地 STT 和 3-* legacy route 里的小米云 ASR 结果可能写法不同，例如：

```text
一加一等于几
1+1等于几
```

如果运行期同时保留监听两条路线，去重时要做 canonical key，避免同一个问题触发两次 Hermes。例如两者都归一成：

```text
1+1=几
```

## 准确率记录

建立一个简单表格：

```text
sample_id | expected | actual | pass | notes
```

通过标准：

```text
短句核心语义正确即可，不要求标点完全一致。
```

失败类型要分类：

```text
漏掉唤醒词
错字但语义可用
核心词识别错误
音频太短
静音
噪声
```

## 速度记录

每次 STT 记录：

```text
audio_duration_ms
stt_duration_ms
realtime_factor = stt_duration_ms / audio_duration_ms
```

第一阶段目标：

```text
3 秒以内短句，STT 耗时 < 2 秒
```

优化目标：

```text
STT 耗时 < 1 秒
```

## 封装接口建议

后续代码可以封装为：

```python
async def transcribe_utterance(audio_path: str, session_id: str) -> dict:
    return {
        "session_id": session_id,
        "audio_path": audio_path,
        "text": text,
        "normalized_text": normalized_text,
        "question": question,
        "duration_ms": duration_ms,
        "engine": engine,
    }
```

这个函数只做 STT，不调用 Hermes。
