# 3.1 步骤：确认 Hermes 调用入口

## 目标

确认 Mac Mini 上的 Hermes 可以被服务侧程序非交互调用。

这是服务侧改造的第一步。只有先确认 Hermes 能稳定返回短回答，后面再把它接进 xiaozhi server 才有意义。

## 当前已知

Hermes CLI 已安装在：

```text
/Users/chenpengfei/.local/bin/hermes
```

当前版本曾检测到：

```text
Hermes Agent v0.13.0
```

## 手动验证命令

在 Mac Mini 上执行：

```sh
hermes chat -Q -q "请用一句话介绍你自己，不要使用 Markdown。"
```

期望：

- 命令能退出。
- stdout 返回一段短文本。
- 不需要进入交互式 TUI。
- 输出适合后续直接交给音箱播报。

## 第一版 server 调用方式

如果上面的命令可用，第一版 Hermes server 可以直接通过 subprocess 调用：

```sh
hermes chat -Q -q "请用适合语音播报的方式回答，控制在 300 字以内，不要使用 Markdown：用户问题"
```

第一版先不追求最低延迟。优先目标是：链路清楚、日志可见、容易排障。

## 回答约束

给 Hermes 的 prompt 应该限制：

- 用中文回答。
- 适合语音播报。
- 不使用 Markdown。
- 不输出代码块。
- 控制在 300 字以内。
- 如果问题不清楚，简短追问。

示例：

```text
请用适合语音播报的中文回答，控制在 300 字以内，不要使用 Markdown，不要输出代码块。用户问题：<问题>
```

## 后续可选方案

CLI 跑通后，如果需要更低延迟或更强会话控制，再评估：

- Hermes API server / gateway API。
- OpenAI-compatible API。
- 常驻 Hermes 子进程。
- 独立 session 管理。

## 成功标准

- 能从 shell 获得 Hermes 短回答。
- 回答文本可以直接传给 `speaker.play(text=...)`。
- 命令失败时能拿到 stderr / exit code，方便 server 兜底播报错误提示。
