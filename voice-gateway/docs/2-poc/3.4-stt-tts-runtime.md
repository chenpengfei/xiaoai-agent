# 3.4 步骤：STT / TTS / 运行方式

本文记录 3-* legacy route 阶段的历史策略。当前主链路已经进入 4-*：音箱本地 KWS/VAD + Mac Mini sherpa-onnx VAD/STT + Hermes + Mac Mini TTS URL + 音箱播放。3-* 只作为 manual rollback 保留。

## 目标

在 `3.3` 的最小 Hermes 闭环跑通后，再逐步优化服务侧体验和运行方式。

本阶段不应该早于最小闭环。否则会同时引入太多变量，排障困难。

## STT 策略

3-* 阶段的历史优先顺序：

1. 如果 xiaozhi server 能拿到小爱原生识别文本，先直接复用。
2. 如果必须处理麦克风音频流，再接 STT。
3. 当时曾考虑用云端 Whisper-compatible API 快速验证。
4. 当时曾考虑稳定后再评估本地 whisper 系列方案。

历史建议：3-* 阶段先不要做本地 STT，先把 Hermes 回答从音箱播出来。当前 4-* 主链路已使用 sherpa-onnx 做本地 VAD/STT。

## TTS / 播放策略

优先顺序：

1. 如果 open-xiaoai 支持让音箱直接播报文本，先用音箱自身播报。
2. 如果需要更自然音色，再由 Mac Mini 生成音频。
3. 如果走音频播放，需要确认 open-xiaoai 下发音频 URL 或音频流的稳定方式。

第一版回答约束：

- 300 字以内。
- 中文口语化。
- 不使用 Markdown。
- 不输出很长列表。

## 打断和音量恢复

最小闭环之后再验证：

- `speaker.abort_xiaoai()` 对当前小爱播报的影响。
- 调用后是否需要等待 1-2 秒让 TTS 恢复可用。
- Hermes 播报时再次唤醒或打断会发生什么。
- 播报结束后音量、播放状态是否恢复。

## 运行方式

开发阶段：

- 前台运行 server。
- 保留详细日志。
- 每次只改一个环节。

稳定后：

- 封装为 Python/Node 常驻进程。
- 再决定是否 Docker 化。
- 最后接入 Mac Mini 开机自启动。

可选自启动方式：

- Docker restart policy。
- macOS launchd。
- 手写 shell 脚本 + 日志目录。

## 长期目标

这些是 3-* 阶段的长期目标；其中本地 STT/TTS 已在 4-* PoC 中验证：

- 本地 sherpa-onnx VAD/STT。
- Mac Mini TTS URL 播放。
- 连续对话。
- 低延迟打断。
- 多轮上下文。
- 多房间音箱路由。
